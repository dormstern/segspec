# segspec: From App Configs to Network Policies in 30 Seconds

## The Problem

Microsegmentation is the #1 recommended defense against lateral movement. CISA mandates it. Every CISO wants it. Yet only 5% of organizations actually enforce it.

Why? Because generating network policies requires three teams to coordinate simultaneously:

- **Security** knows what should be locked down but can't read Kubernetes manifests
- **Network** can write firewall rules but doesn't know what the app connects to
- **App developers** know their dependencies but won't touch network configs

The current workflow looks like this:

1. Deploy your app with zero network restrictions (everything open)
2. Install a traffic observation agent on every pod
3. Wait 2-4 weeks to capture "normal" traffic patterns
4. Have a network engineer manually translate flow logs into NetworkPolicy YAML
5. Apply in audit mode for another 2 weeks
6. Pray you didn't miss anything, then flip to enforce
7. Something breaks at 2 AM. Roll back. Start over.

**Total time: 4-8 weeks per application. Cost: $50K+ in engineering hours per app.**

Most teams give up after step 3.

## What segspec Does

segspec reads your application's config files and generates Kubernetes NetworkPolicy YAML. No agents. No runtime access. No observation period.

It understands:

- Spring Boot configs (application.yml, application.properties)
- Docker Compose files
- Kubernetes manifests (Deployments, Services, ConfigMaps)
- Environment files (.env)
- Build files (pom.xml, build.gradle)

For each file, it extracts every network dependency — database connections, cache endpoints, message brokers, API URLs, service-to-service calls — and outputs a ready-to-apply NetworkPolicy.

## How to Use It

### Who runs it

**You.** The security engineer, the platform engineer, the DevOps lead, the SRE — whoever needs network policies. You don't need the app developer's help. You don't need network team approval to generate the policy. You just need access to the app's source code repository.

### Where it runs

**On your laptop.** segspec is a single binary with zero dependencies. It reads files from disk. It doesn't need cluster access, cloud credentials, or a running application. It works offline.

### What you need

Access to the application's Git repository. That's it. If you can `git clone` the repo, you can generate its network policies.

### Step-by-step

**1. Install segspec (once)**

```bash
# macOS / Linux
curl -fsSL https://github.com/dormstern/segspec/releases/latest/download/segspec-$(uname -s | tr '[:upper:]' '[:lower:]')-$(uname -m) -o /usr/local/bin/segspec
chmod +x /usr/local/bin/segspec
```

Or with Go:

```bash
go install github.com/dormorgenstern/segspec@latest
```

**2. Clone the app repo you want to secure**

```bash
git clone https://github.com/your-org/your-app.git
```

If you already have the repo checked out — skip this step.

**3. Run segspec**

```bash
segspec analyze ./your-app
```

Output:

```
Service: your-app
Dependencies: 12

  -> postgres-primary:5432/TCP  [high]  PostgreSQL
    source: your-app/src/main/resources/application.yml
  -> redis-cache:6379/TCP       [high]  Redis
    source: your-app/docker-compose.yml
  -> kafka-broker:9092/TCP      [high]  Kafka
    source: your-app/src/main/resources/application.yml
  -> user-service:8080/TCP      [high]  HTTP service
    source: your-app/k8s/deployment.yaml
  ...

Confidence: 10 high, 1 medium, 1 low
```

Every dependency listed with its port, protocol, confidence level, and the exact file where it was found.

**4. Generate NetworkPolicy YAML**

```bash
segspec analyze ./your-app --format netpol > network-policy.yaml
```

Output is valid Kubernetes NetworkPolicy YAML:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: your-app-egress
  labels:
    generated-by: segspec
spec:
  podSelector:
    matchLabels:
      app: your-app
  policyTypes:
    - Egress
  egress:
    - to: # postgres-primary
      ports:
        - port: 5432
          protocol: TCP
    - to: # redis-cache
      ports:
        - port: 6379
          protocol: TCP
    - to: # kafka-broker
      ports:
        - port: 9092
          protocol: TCP
    # ... all dependencies listed
    - to: # DNS (always included)
      ports:
        - port: 53
          protocol: UDP
        - port: 53
          protocol: TCP
```

**5. Apply the policy**

Preview first:

```bash
kubectl apply -f network-policy.yaml --dry-run=client
```

Then apply:

```bash
kubectl apply -f network-policy.yaml
```

Or commit to your GitOps repo and let ArgoCD/Flux handle it:

```bash
cp network-policy.yaml k8s/policies/
git add k8s/policies/network-policy.yaml
git commit -m "add network policy generated by segspec"
git push
```

**6. (Optional) AI-enhanced analysis**

For deeper analysis, add the `--ai` flag. segspec sends your config files to Claude and asks it to find dependencies the rule-based parsers might have missed:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
segspec analyze ./your-app --format all --ai
```

AI-discovered dependencies are marked with `[AI]` prefix and medium confidence so you know to verify them.

## Running in CI/CD

Add segspec to your pipeline to generate policies on every merge:

```yaml
# GitHub Actions
- name: Generate Network Policies
  run: |
    curl -fsSL https://github.com/dormstern/segspec/releases/latest/download/segspec-linux-amd64 -o segspec
    chmod +x segspec
    ./segspec analyze . --format netpol > k8s/network-policy.yaml

- name: Commit policy
  run: |
    git add k8s/network-policy.yaml
    git diff --cached --quiet || git commit -m "update network policy [segspec]"
```

Now your network policies stay in sync with your code automatically.

## FAQ

**Q: Does segspec need access to my running cluster?**
No. It reads config files from disk. It never connects to anything (unless you use `--ai`, which calls the Claude API).

**Q: What if my app uses environment variables that are only set at runtime?**
segspec extracts what's in the config files. If your `application.yml` says `spring.redis.host: ${REDIS_HOST}`, segspec can't resolve that placeholder. But if your Docker Compose or K8s manifest sets `REDIS_HOST=redis-cache`, segspec will find it there. The `--ai` flag can also infer likely values from context.

**Q: What about Helm templates?**
Not yet supported in v1.0. Helm template resolution (e.g., `{{ .Values.redis.host }}`) is on the roadmap. For now, run `helm template` first to render the YAML, then point segspec at the output.

**Q: Can it generate per-service policies instead of one big policy?**
v1.0 generates a single aggregate policy. Per-service policies (one NetworkPolicy per source service with proper ingress/egress) are coming in v1.1.

**Q: What if the generated policy is wrong?**
Every dependency includes a confidence level and the exact source file. Review the summary output before applying. Start with `--dry-run=client` to see what would change. Low-confidence dependencies get a warning so you know to verify them.

**Q: Does it work with non-K8s environments?**
The analysis works anywhere — segspec reads config files regardless of your runtime. But the output is currently K8s NetworkPolicy YAML only. AWS Security Groups, Azure NSGs, and Cilium policies are on the roadmap.

**Q: How is this different from Calico/Cilium flow log analysis?**
Flow log analysis observes runtime traffic after deployment (reactive). segspec reads config files before deployment (proactive). They're complementary — use segspec to generate the initial policy, then use flow logs to validate it in production.
