# Changelog

All notable changes to segspec are documented in this file. This project adheres
to [Semantic Versioning](https://semver.org/).

## v0.6.0-dev

- **Forgiving `--format` aliases with deprecation warnings** — eight common alternate spellings of `--format` values now resolve to their canonical form so first-time users no longer hit "unknown format" on a near-miss spelling. Each aliased run emits a single `Warning: --format <alias> is deprecated, use --format <canonical>` line to stderr (warnings never touch stdout, so rendered YAML stays pipeable into `kubectl apply`). Recognized aliases: `networkpolicy` and `network-policy` → `netpol`; `audit-ledger` → `audit`; `default-deny-only` → `default-deny`; `evidencebundle` and `evidence_bundle` → `evidence-bundle`; `cilium-network-policy` and `cnp` → `cilium`. Matching is case-insensitive. Lookup lives in a new `internal/formats` package so future format renames have a single hook to add the alias + warning, rather than scattering string compares through the dispatch. Free tier.
- **Workload coverage report (`segspec coverage <path>`)** — new top-level subcommand that cross-checks the workloads declared in app configs / Kubernetes manifests against the `NetworkPolicy` / `CiliumNetworkPolicy` YAML in the same path. Answers two operator questions in one shot: which workloads have NO matching policy, and which policies select zero workloads (orphan policies). Output is a human-readable table by default, or `--json` for CI ingestion. The report itself is free tier; the `--exit-code` CI gate (with `--threshold N` to relax the default 100% bar) is gated behind a Pro license, mirroring `diff --exit-code`. Cited in landscape.md E-005 (Tigera blog, "policies often overwhelm ordinary and veteran users") and features.json `policy-coverage-report` priority 9: the auditor's first question, answered in one command.
- **Policy stack explainer (`segspec explain <workload> --policies <path>`)** — new top-level subcommand that takes a workload name + labels and a directory of NetworkPolicy / CiliumNetworkPolicy / CiliumClusterwideNetworkPolicy YAML, finds every policy that selects the workload (via `podSelector` / `endpointSelector` matchLabels), and prints the union of contributed allow rules — the workload's effective allow-set — with `file:line` evidence per rule. Models the additive K8s semantic explicitly: declaring ANY ingress (or egress) rule on a selecting policy flips that direction from allow-by-default to deny-by-default, even if the rule list is empty. Default output is human-readable Markdown grouped by applied-policy then effective-set; `--json` emits a structured contract (`{workload, policies[], effective_ingress[], effective_egress[], default_deny_ingress, default_deny_egress}`) for tooling. Reuses the wave-2 `internal/parser/netpol` adapter so the same YAML the validator lints is the YAML the explainer reads — no second parser surface, no drift. Cited from cilium/cilium#42904 ("evaluation order for network policy rules ... ClusterNetworkPolicy / Kubernetes NetworkPolicy / CiliumNetworkPolicy"): upstream maintainers couldn't agree on semantics, segspec describes what's actually there. Free tier.
- **Cilium-native output (`--format cilium`)** — emits a `CiliumNetworkPolicy` (`apiVersion: cilium.io/v2`) anchored on `endpointSelector` instead of vanilla `podSelector`, routing each dependency to the right Cilium peer shape: in-cluster service names → `egress[].toEndpoints[]`, external hostnames → `egress[].toFQDNs[].matchName` (or `matchPattern` for wildcards like `*.amazonaws.com`), and literal IPs → `egress[].toCIDR[]`. When any `toFQDNs` rule is emitted, segspec also emits a paired DNS egress block to `kube-system`/`kube-dns` on UDP+TCP 53 with a `dns.matchPattern: "*"` rule — exactly what the validator's `missing-dns-egress` check (cilium/cilium#44504) demands, so segspec never emits a policy that fails its own linter. Inherits the wave-2 disable semantics: deps with `Disabled="egress"` or `"full"` are skipped. Free tier. Driven by issue-mining where 6 of 8 cycle-1 authoring-pain quotes came from Cilium operators (research/issue-mining-2026-05-04.md), and the README roadmap already promised native CNP output.
- **Source-level disable annotation (`# segspec:disable=ingress|egress|full` / `segspec.io/disable`)** — operators can now temporarily opt a workload out of policy generation by leaving an inline comment in the source config (Compose, Spring `application.yml`/`.properties`, K8s, `.env`, `pom.xml`, `build.gradle`) or by stamping the `segspec.io/disable` label/annotation on a Kubernetes Deployment/StatefulSet. The directive is surfaced verbatim in `--format summary` output (`[disabled: ingress]`) and in the audit + evidence-bundle ledgers — auditors WANT to see suppressed rules — but `--format netpol`, `per-service`, and `default-deny` skip emitting the corresponding rule(s). Closes the authoring-pain gap reported at kubernetes/kubernetes#112560: "Sometimes, I want to disable the networkpolicy temporarily. Now, I must delete it or edit it to match none." K8s upstream rejected the field-level fix as a spec change; segspec solves it at the source-of-truth layer instead. Free tier.
- **Bundled demo fixtures (`segspec analyze --demo <name>`)** — two embedded, license-clean fixture corpora ship inside the binary so first-time users can reproduce a realistic dependency graph without first finding a public repo to point segspec at. `--demo list` enumerates the catalog; `--demo sentry-mini` analyzes a synthesized ~17-service Sentry-style Compose+Helm+.env stack; `--demo microservices-demo` analyzes a 6-service adaptation of Google's Apache-2.0 microservices-demo. The flag is mutually exclusive with the positional `<path>`. Cited from landscape.md §0: "the bottleneck is conversion, not discovery — ~50k aggregate-star visibility yielding 15 stars implies the README hook / first-impression / wedge framing is leaking traffic." This is the cheapest first-impression fix in the wave-1 backlog. Implemented via `embed.FS` + materialize-to-tempdir adapter (~50 LOC) so the existing walker stays untouched. Free tier.
- **Policy validator (`segspec validate <path>`)** — new top-level subcommand that lints existing Kubernetes `NetworkPolicy` and Cilium `CiliumNetworkPolicy` / `CiliumClusterwideNetworkPolicy` YAML for the four operator-reported failure modes the apiserver does not catch. Quoting cilium/cilium#44504 verbatim: "you can create valid network policy that is not rejected by apiserver and that are logically incorrect and do not work: 1. toFQDNs without specifying dns rules in egress block for dns service. 2. toEntities with toPorts." Checks: `missing-dns-egress` (cilium #44504), `to-entities-with-to-ports` (cilium #44504), `oversized-selector-label` (cilium #43771), `unreferenced-selector` (best-practice). Each finding carries `file:line` evidence and the upstream issue URL. Plain-text or `--json` output; exits 1 on any error-severity finding so CI fails the build. Free tier — distinct from `analyze`/`diff`/`snapshot` (those read application configs; `validate` reads NetworkPolicy YAML).
- **Provenance-stamped baselines (`segspec snapshot`)** — new top-level subcommand writes a baseline JSON enriched with a `metadata` block (`created_utc`, `segspec_version`, `git_commit`, `git_dirty`, `input_path`). `segspec diff` accepts both the new shape and the legacy bare-`DependencySet` shape (no metadata) — backward compatible by design. When the baseline's `segspec_version` differs from the running binary, diff emits a one-line warning to stderr so version-skew shows up in CI logs. Cited from features.json `policy-baseline-snapshot` (priority 10): IDENTITY.md secondary-persona JOB ("review and approve segmentation in CI instead of after an incident") requires baseline as a stable, audit-trail artifact rather than an ad-hoc JSON file.
- **Default-deny scaffold output (`--format default-deny`)** — emits a namespace-scoped default-deny `NetworkPolicy` plus all per-service allow policies in one multi-document YAML, satisfying the auditor demand documented at https://github.com/cilium/cilium/issues/43502 ("cluster auditors want every component to have a targeted NetworkPolicy"). Free tier.
- **Audit ledger output (`--format audit`)** — auditor sign-off Markdown that groups every dependency by workload, separates ingress/egress, and ends with a control-mapped checklist. Driven by cluster-auditor authoring pain in https://github.com/cilium/cilium/issues/43503 ("cluster auditors require a defined networkPolicy for all deployments/daemonsets/etc.").
- **Offline license-key validation (`--license-key`)** — paid features now require an Ed25519-signed JWT license. The validator runs fully offline against a public key compiled into the binary; no network calls, no phone-home, no telemetry. Pass the token via `--license-key <token>` or the `SEGSPEC_LICENSE_KEY` environment variable. Issued at https://segspec.dev/pro.
- **Gated paid features** — `--format evidence`, `--format per-service`, and `diff --exit-code` now require a Pro (or Enterprise) license. Free formats (`summary`, `netpol`, `json`) remain free forever. Running a gated feature without a valid license exits with code 2 and a clear error pointing at https://segspec.dev/pro.
- **Per-parser version stamping** — every parser (spring, compose, k8s, envfile, buildfile) now exposes a `Version` constant (currently `0.6.0`), and `parser.Versions()` returns the full format → version map. Required for evidence-bundle reproducibility: pin a baseline to a parser-version set and consciously accept upgrades.
- **BREAKING_CHANGE:** JSON output now includes a top-level `"parser_versions"` field. Existing tooling that parses segspec JSON should ignore unknown top-level fields. The empty-result shape changed from `{}` to a populated object containing `parser_versions` and `dependencies: []`. Reported `version` field bumped from `0.5.0` to `0.6.0`.
- **Auditable evidence bundle export (`--format evidence-bundle`, `--format evidence-bundle-sarif`)** — emits a deterministic, signed-shape JSON containing every dependency, file/line/declaration evidence block, segspec version, parser-version map, and an `input_tree_sha256` fingerprint of the analyzed config tree (sorted-by-path, content-sensitive, order-insensitive). The SARIF v2.1.0 companion wraps the same evidence as one `result` per dependency at level `note` with a `physicalLocation.artifactLocation`, ready for ingestion by GitHub code-scanning, Defender, and Splunk. Closes the AppSec evidence-trail gap left by RHACS roxctl's deprecation (CL-001) and directly serves IDENTITY.md secondary persona ("evidence that its NetworkPolicy matches its declared dependencies, so I can review and approve segmentation in CI"). Pro tier.
- Bumped reported version from `0.5.0` to `0.6.0-dev`.
